# ESR-002 – Minimum Viable Platform (MVP-001)

## Status

Approved

Mode: Documentation-only

---

## Objective

Define the first operational version of AlpenTind that validates Enterprise Architecture through real operational use.

MVP-001 is operational-first validation through one complete real Journey workflow from Activity Design to Execution.

---

## Mandatory References

This ESR shall conform to:

- ADR-003 Enterprise Architecture Standard
- PDR-011 Platform Architecture
- PDR-012 Reference Domain
- PDR-013 Activity Design
- PDR-014 Execution
- ESR-001 Engineering Implementation Standard

This ESR does not redefine architecture, ownership, or domain boundaries established by these references.

---

## Design Directive (Critical)

- This ESR defines MVP implementation scope and constraints, not new architecture.
- No new Business Domains may be introduced.
- No new Business Objects may be introduced.
- Scope may not be broadened for feature completeness.
- ADR-003 and ESR-001 compliance is mandatory and may not be relaxed.

---

## 1. Operational Goal

MVP-001 shall make one complete real Journey workflow operationally usable.

A certified guide shall be able to:

1. Create Activity Design
2. Design using Reference Domain knowledge
3. Professionally approve Activity Design
4. Create Journey
5. Activate Journey
6. Execute via Operational Commitments
7. Complete operational work in platform

Normative requirement: the system must be operationally usable in real work.

---

## 2. Included Scope

Only capabilities required to operate one real Journey end-to-end are included.

### 2.1 Included Business Domains

- Reference Domain
- Activity Design
- Journey
- Execution

### 2.2 Included Platform Views

- Follow-up (minimum interface required for operational work)

### 2.3 Included Business Objects

- Reference Objects
- Selection Objects
- Journey
- Operational Commitments

Constraint: no additional Business Objects.

### 2.4 Included Business Events

- Professional Approval
- Journey Creation
- Activate Journey

Publication may be manual in initial MVP operation.

---

## 3. Required Functional Capabilities

### 3.1 Reference Domain

- Create Areas
- Create Accommodation
- Create Trails

### 3.2 Activity Design

- Create Activity Design
- Select Reference Objects
- Maintain Selection Objects
- Perform Professional Approval (human)

### 3.3 Journey

- Create Journey
- View Journey
- Activate Journey

### 3.4 Execution

- Generate Operational Commitments
- Update commitment status
- Add operational comments
- Maintain deadlines
- Complete commitments

### 3.5 Follow-up View

- Display commitments requiring attention
- Display attention reason
- Navigate directly to affected commitment

---

## 4. Included vs Excluded Matrix

| Area | Included in MVP-001 | Excluded in MVP-001 |
|---|---|---|
| Business Domains | Reference Domain, Activity Design, Journey, Execution | Any additional business domain |
| Platform Views | Follow-up | Overview, Calendar, My Workday |
| Business Objects | Reference Objects, Selection Objects, Journey, Operational Commitments | Any additional business object |
| Business Events | Professional Approval, Journey Creation, Activate Journey | Any additional event not required for one real Journey flow |

---

## 5. Explicitly Excluded (Strict)

Do not implement:

- Authentication / Authorization
- Customer Portal / Website
- Booking Engine
- Payments
- Email / SMS / Notifications
- Calendar / Overview / My Workday
- Reporting / Analytics
- AI capabilities
- External integrations
- Automation / scheduling engine
- Mobile optimization / offline support
- Advanced search

---

## 6. Operational Validation

MVP-001 validation occurs through real AlpenTind operations:

- real Journey creation and execution on platform
- real operational use by professionals
- feedback from operations drives future scope
- architecture changes only via ADR/AVR governance

---

## 7. Success Criteria

MVP-001 is successful when the platform can:

- create real Activity Design
- approve Activity Design
- create Journey
- activate Journey
- generate Operational Commitments
- execute and complete commitments
- operate one real Journey end-to-end

---

## 8. Engineering Constraints and Architecture Compliance

Implementation must comply with ADR-003 and ESR-001.

The following constraints are mandatory:

- Business Domains own business truth
- Platform Views own no business data
- Business Objects have single owner
- Operational Commitments are generated by Journey activation
- no enterprise-architecture violations

---

## 9. Future Expansion (Deferred Post-MVP)

The following are explicitly deferred:

- Overview / Calendar / My Workday
- Customer Management
- Website Integration
- Booking Management
- Partner Integrations
- Payments
- Automation
- AI
- Operational Analytics

Rule: expansion is allowed only after MVP-001 operational validation.

---

## Definition of Done

- one complete operational workflow defined
- enterprise architecture preserved
- one real Journey can be executed
- Operational Commitments fully functional in defined scope
- Follow-up view operational in defined scope
- platform ready for real operational validation
